# Project Instructions

이 프로젝트를 수정할 때는 아래 원칙을 우선한다.

## 프로젝트 방향

- 거창한 SaaS 랜딩페이지처럼 만들지 않는다.
- AI가 만든 것 같은 과장된 문구를 쓰지 않는다.
- 작고 간단한 무료 도구 사이트처럼 보이게 한다.
- 사용자는 설명을 읽으러 온 것이 아니라 바로 쓰러 온다.
- 기능은 적어도 좋으니, 실제로 편하게 작동해야 한다.

## 문구 원칙

- "한국형", "포털", "혁신적", "강력한", "올인원", "최고의" 같은 표현을 피한다.
- 짧고 건조하게 쓴다.
- 사용자가 얻는 결과를 바로 말한다.
- 개발자 용어는 메인 페이지에서 최소화한다.
- "브라우저에서 처리합니다" 정도는 괜찮지만 File API, Canvas, localStorage 같은 단어는 privacy나 상세 설명에서만 쓴다.

## 디자인 원칙

- 히어로 영역을 크게 만들지 않는다.
- 첫 화면에서 검색창과 자주 쓰는 도구가 보여야 한다.
- 카드 그림자, 배지, 색상을 과하게 쓰지 않는다.
- 여백은 충분히 주되 비어 보이지 않게 한다.
- 모바일에서 먼저 편해야 한다.
- 버튼은 구체적으로 쓴다. 예: "도구 열기" 대신 "개인정보 가리기", "PDF 정리하기".

## 기능 원칙

- 아직 구현되지 않은 기능은 메인에 크게 노출하지 않는다.
- ready/beta/planned 상태를 구분한다.
- ready 또는 beta 도구만 메인에 보여준다.
- planned 도구는 /tools 하단의 "준비 중인 도구"로 분리한다.
- 파일이나 텍스트를 서버로 보내는 코드를 만들지 않는다.
- 핵심 기능은 브라우저에서 처리한다.

## 품질 기준

- 변경 범위를 요청받은 작업으로 제한한다.
- 불필요한 대규모 리팩터링을 하지 않는다.
- TypeScript 오류가 없어야 한다.
- `npm run build`가 통과해야 한다.
- 가능한 경우 `npm run test`도 통과해야 한다.
- 작업 후 수정 파일, 변경 내용, 테스트 결과를 짧게 보고한다.
